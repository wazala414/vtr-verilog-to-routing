ZeroAMP Project

Transistors are the building blocks of digital integrated circuits found in every modern electronic device, from cell phones to industrial electronics.
However, more and more emerging applications utilise ultra-low power sources and operate at low power in harsh environmental conditions where transistors cannot perform efficiently.

In ZeroAMP, we will address the societal and industrial needs for such computation solutions by pushing beyond the limits of transistors.
To tackle this challenge, we will develop novel nanoelectromechanical (NEM) switches that survive extreme environments.

The ZeroAMP Suite focuses on the benchmarking of a NEM-based FPGA architecture, including the implementation of the switchblock matrix.
This suite will also include a programming feature for our future NEM-based FPGA architecture.

Verilog-To-Routing (VTR)

Verilog-To-Routing also known as VTR is an open-source suite of tools for benchmarking FPGA architectures. The suite takes as input architecture descriptions and verilog description of circuit to be mapped on the given architecture.
VTR is a complete flow that is able to simulate an architecture, optimize a circuit description to best map onto the architecture, place, pack and route the circuit and finally perform a performance analysis.
For more information on VTR you can access the documentation by clicking here.
The GUI allows the user to easily execute simple analysis using VTR. The VTR section of the GUI is split in 3 subsections: Single, Batch and Parse.

The Single subsection allows to quickly process simple task providing the following parameters:

 Architecture: Describes the FPGA architecture from a XML file.
 Circuit: Describes the circuit to map on the architecture from a verilog file.
 Output Directory: Location of all the output files from VTR.
 Routing Width: Specifies the number of channel/tracks in connect blocks.
 Analysis: Performs an analysis when displaying the graphical representation. Unticking allows incomplete task to still display.
 [optional] Args: List extra arguments. Refer to the command-line options of the VTR Documentation for more information on supported arguments.

The Run button executes the regular VTR flow using the specified parameters and save the output at the desired location.
On the other hand, the Display will display the graphical representation using the previously saved outputs.

The Batch subsection allows to process a list of task providing the following parameters:

 Task List: Describes the task list from a text file. To create a task and task lists refer the to the relevant section in the Task section of the VTR Documentation.
 Short Names: Shortens the folder names when processing a task list. Unticking will make the folders contain the full description of all arguments of a task.
 [optional] Args: List extra arguments. Refer to the command-line options in the VTR Documentation for more information on supported arguments.

The Run button executes the VTR task flow using the specified parameters.
All the outputs will be saved at the location of the task configuration files.

The Parse subsection allows to quickly parse/re-parse results from previously processed tasks providing the following parameters:

 [for batch] Task List: Describes the task list from a text file. To create a task and task lists refer the to the relevant section in the Task section of the VTR Documentation.
 [for single] Parse Parameters: Describes the regex list for parsing the results in a table.
 [for single] Output Directory: Location of the input files from VTR when parsing a single task. This will also be the location of the parsed files.
 Batch: Enables the parsing of multiple task using a task list.
 [optional] Args: List extra arguments. Refer to the parse flow and the parse task sections of the VTR Documentation for more information on supported arguments of the single execution and batch execution respectively.

The Parse button executes the parse flow for single task or tasks from a task list according to the parameters entered.
This option can be used when the variables of interest extracted from the output of VTR are no longer relevant.
Changing the content of the parse parameter file then allows to extract the new relevant varibles from already processed outputs.

Routing Export

The routing export section of the GUI allows the user to use the routing path output from VTR and generate a database of the pin-to-pin connection of all switchblocks.
This option is used for describing the required patterns for switchblock matrix based FPGA architecture and needs to be used in conjunction with the programming section.
To run this script the following parameters will be used:

 Routing Outcome: The routing ouput file of VTR using the extension .route.
 Database Output Folder: Specify the location of the output CSV database.
 Name: Specify the name of the output CSV database.
 [optional] Regex Output: When ticked the program exports a detailed file of the regex matches according to the file name specified.

The Run button executes the script that returns a database describing the pin-to-pin patterns for a list of all the switchblock in the architecture.

Programming

The programming section of the GUI is closely linked to the routing export section.
It is responsible for computing NEM orientation required in all switchblock matrix of an FPGA architecture to achieved the connection patterns identified from the routing export section.

 Routing Outcome: The routing ouput file of VTR using the extension .route.
 Database Output Folder: Specify the location of the output CSV database.
 Name: Specify the name of the output CSV database.
 [optional] Regex Output: When ticked the program exports a detailed file of the regex matches according to the file name specified.

The Run button starts the creation of the CSV database.
From the output produced, the user can use the information to configure a NEM-based FPGA.
